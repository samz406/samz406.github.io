4种架构介绍

发表于 2025-04-14 分类于 架构

在构建现代企业级系统的过程中，“架构”二字已不仅仅代表软件模块的堆叠或技术选型的权衡。真正成熟的架构设计，往往需要在业务目标、应用协作、数据支撑、技术实现之间，建立一种系统性、全局性的秩序。而TOGAF（The Open Group Architecture Framework），作为业界广泛认可的企业架构方法论，正是为了解决这一多层次复杂性的挑战而生。

TOGAF将企业架构拆解为四个核心维度：业务架构、应用架构、数据架构、技术架构。这四个层次各自承担着不同的职责，彼此之间既有明确边界，又相互耦合、协同进化。

业务架构：对齐战略与能力的桥梁

✦ 定义与职责

业务架构关注的是企业“做什么”以及“为什么这么做”。它负责梳理企业的核心业务能力、组织结构、流程流程、角色分工，以及它们之间的逻辑关联。

换句话说，业务架构是从业务战略到系统需求之间的翻译器。没有清晰的业务架构，系统设计只能靠拍脑袋，而不会真正服务于企业目标。

✦ 关注核心
  • 企业愿景与战略目标
  • 关键业务流程与场景（如“下单-支付-发货-售后”）
  • 业务能力建模（例如“会员管理”、“积分体系”、“商品目录”等）
  • 用户角色与组织结构（角色权限、协同路径）
  • 关键业务场景的生命周期建模
✦ 落地产出
  • 业务能力地图（Business Capability Map）
  • 业务流程图（BPMN流程、活动图等）
  • 角色与责任矩阵（RACI图）
  • 战略目标对能力的映射关系
✦ 实践建议

很多架构师跳过这一层，直接进入“技术架构”或“服务拆分”阶段，这往往是导致后续系统重构频繁的根源。

优秀的业务架构，需要有产品视角的理解力，也需要有对组织政治、战略目标的洞察力。它并非纸上谈兵，而是为整个系统设计提供“方向性锚点”。

应用架构：组织系统协作的骨架

✦ 定义与职责

应用架构关注的是企业中有哪些系统，这些系统之间如何协作，共同支撑业务目标的实现。

它是承接业务架构与技术架构的中间层：上接业务流程与功能需求，下接系统设计与技术实现。好的应用架构不仅回答“要实现什么功能”，还要解决“这些功能应该由哪些系统来实现”、“系统间如何通信”、“数据如何在系统间流转”。

✦ 关注核心
  • 应用清单（系统视角）
  • 系统之间的集成关系（同步/异步，直连/API/消息等）
  • 应用责任边界与耦合度
  • 关键系统的接口协议、服务目录
  • 系统生命周期与演化策略（替换、整合、沉淀）
✦ 落地产出
  • 应用架构图（Application Landscape Diagram）
  • 系统边界图与集成图（Context Diagram、C4模型）
  • 接口定义文档（OpenAPI、异步消息契约）
  • 服务注册与发现机制设计
  • 应用治理机制（如系统分级、稳定性等级）
✦ 实践建议

在微服务化、SaaS化趋势日益明显的今天，应用架构越来越不是“系统清单”，而是“系统网络”。

现代应用架构关注的问题不再是“有没有系统支持”，而是“系统组合效率高不高”、“系统协作代价大不大”、“变更是否可控”。这要求架构师具备较强的横向协作建模能力，以及对服务划分原则（如 DDD）有深刻理解。

数据架构：支撑业务认知与决策的血液系统

✦ 定义与职责

数据架构关注的是企业的数据资产如何建模、存储、流动与治理。它承担着连接“业务实体”与“技术存储”的职责，是系统中最具“长期价值沉淀”的一层。

在数字化浪潮中，企业数据已不仅仅是运行的副产物，而成为重要的战略资源。数据架构的好坏，直接影响企业数据质量、使用效率、以及后续的分析与AI能力。

✦ 关注核心
  • 关键业务对象的数据建模（如订单、用户、交易、商品）
  • 数据实体之间的关系与血缘
  • 数据分层（ODS、DWD、DWS、ADS等）
  • 数据主权与生命周期管理
  • 数据安全、脱敏、合规治理
✦ 落地产出
  • 数据实体关系图（ER模型、逻辑模型）
  • 数据血缘图、数据流图
  • 数据标准与命名规范
  • 数据字典、主数据管理模型
  • 数据访问与权限控制策略
✦ 实践建议

很多系统只关注“接口”和“功能”，却忽略了数据模型的抽象与统一，导致数据冗余、命名混乱、难以沉淀与复用。一个系统生命周期越长，数据架构的重要性就越突出。

现代数据架构不应只围绕数据库建模，而要融入数据仓库建设、流式处理、数据中台等理念，做到数据的“可用、可信、可控、可分析”。

技术架构：系统能力与非功能诉求的支点

✦ 定义与职责

技术架构关注的是系统如何实现、部署、运行和演化的技术手段与基础设施选择。

它是企业IT能力落地的核心所在，负责承接上层的应用与数据需求，并提供支撑性能力，如计算资源、网络通信、安全策略、开发框架等。

✦ 关注核心
  • 技术选型与栈结构（如Spring Boot、Vue、K8s、Redis、Kafka等）
  • 服务部署策略（多活、灰度、弹性伸缩）
  • 非功能需求设计（可用性、可扩展性、可维护性、安全性）
  • 中间件能力（缓存、MQ、网关、服务注册）
  • 自动化运维（CI/CD、监控、告警）
✦ 落地产出
  • 技术栈清单与框架规范
  • 系统部署架构图（如三层架构、微服务、Serverless）
  • 监控体系（如Prometheus + Grafana）
  • 灰度发布与容灾策略
  • DevOps 工程流水线设计
✦ 实践建议

技术架构往往是开发团队最熟悉、最常讨论的一层，但也最容易“脱离业务”、走向技术堆砌。

优秀的技术架构并非“新潮”，而是能与上层应用、数据形成良性闭环，为业务演进提供稳定、敏捷、安全的支撑力。

架构不是分层，而是协作

TOGAF 提供了一个清晰的四层视角，帮助我们从战略到技术，从业务到代码，有序地搭建企业系统。然而，这四层架构并不是“谁在上谁在下”的分层结构，而是一种“多维协作、动态反馈”的机制。

  • 业务架构提供方向与目标；
  • 应用架构组织系统实现与协作；
  • 数据架构打通信息流，支撑认知；
  • 技术架构提供基础能力，确保落地可行。

架构设计的终极目标，不是追求某种“完美模型”，而是持续地用最小成本驾驭最大复杂性，让系统在业务变化、技术演进、组织调整中保持韧性与秩序。

在这个意义上，真正优秀的架构师，不仅能建图纸、定规则，更能在混沌中抽象出秩序，在演化中保留灵活的支点。
