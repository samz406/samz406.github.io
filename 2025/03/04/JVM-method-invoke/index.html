JVM方法调用

发表于 2025-03-04 分类于 JVM

方法调用是程序运行中最常见、最核心的操作之一。无论是简单的main方法调用，还是复杂的框架内部反射调用，对JVM而言，都需要处理好不同类型的方法调用指令、维护好调用栈、处理动态链接等诸多细节，以保证程序的正常执行

JVM的核心目标之一就是实现“一次编写，到处运行（Write Once, Run Anywhere）”的理念。Java代码在编译阶段被编译器（如 javac）转为与平台无关的字节码（*.class 文件），随后由JVM在不同平台上加载和执行。JVM是一种基于栈的虚拟机，它使用栈帧（Stack Frame）、操作数栈（Operand Stack）和局部变量表（Local Variable Table）来执行字节码指令。

类加载：方法调用的前奏

类的生命周期

JVM的方法调用始于类加载阶段。当一个类被加载时，JVM会经历加载、验证、准备、解析和初始化五个阶段。其中，解析阶段会将常量池中的符号引用转换为直接引用，为方法调用做好准备。

方法表

每个类在JVM中都有一个方法表。比如Object类的方法表包含hashCode()、equals()等方法的直接地址。子类会继承父类的方法表，并在需要时覆盖或扩展。

静态绑定与动态绑定

JVM中有两种绑定方式：

  • 静态绑定 ：编译期确定（如private、static方法）
  • 动态绑定 ：运行期确定（如虚方法）

当调用animal.eat()时，实际执行的方法取决于运行时对象的类型，这就是动态绑定。

字节码指令

JVM运行时数据区

JVM规范把运行时数据区分成以下几个主要部分：

  1. 程序计数器（PC寄存器）：每个线程都有一个程序计数器，用于记录当前所执行字节码的行号指示器。如果正在执行的是本地方法（native），则PC寄存器值是未指定的。
  2. Java虚拟机栈：每个线程都会创建一个私有的栈，用于存放栈帧。每一次方法调用时都会在该栈中压入一个新的栈帧，方法执行结束（或抛出未处理异常）后栈帧弹出。
  3. 本地方法栈：与Java虚拟机栈类似，只不过它为本地方法服务。它的具体结构和实现方式因不同JVM厂商而异。
  4. 堆：Java中所有对象实例以及数组都要分配在堆上，几乎是整个JVM中占用内存最大的区域。它是所有线程共享的，可以被垃圾收集器统一管理。
  5. 方法区：也是所有线程共享的一个区域，用来存储已被加载的类信息、常量、静态变量和JIT编译器生成的代码等数据。方法区在某些JVM实现中也被称为永久代（PermGen）或元空间（Metaspace）。
  6. 运行时常量池：属于方法区的一部分，每个类或接口对应的常量池会存放它的字面量（如字符串常量）以及符号引用（如类、方法、字段的符号引用）等。这些符号引用在类加载的解析阶段会被解析为直接引用（如内存地址、偏移量等），也就是动态链接的一部分。

以上这些区域对方法调用都有或多或少的影响，其中最重要的就是Java虚拟机栈和方法区中的运行时常量池，因为字节码执行依赖栈帧，而方法调用指令通常需要通过常量池索引来找到需要调用的方法符号引用，然后再解析成可以直接调用的内存地址或方法句柄（Method Handle）。

JVM是基于栈的虚拟机。Java代码编译成字节码后，会产生一系列的指令集。例如访问字段的getfield、给字段赋值的putfield、调用方法的invokevirtual等指令。

在执行某条字节码指令之前，JVM中的解释器（若处于解释执行模式）或者即时编译器（若有编译优化）都会取出当前线程的PC寄存器指示的字节码，然后根据指令含义，操作当前栈帧中的操作数栈或局部变量表，或跳转到新的字节码地址。对于方法调用相关的指令来说，其最典型的过程包括：

  1. 读取指令操作码。
  2. 读取操作数（常量池索引或其他附加数据）。
  3. 根据常量池索引到方法符号引用，随后解析出具体要调用的方法。
  4. 建立新的栈帧（或通过JIT编译器内联、优化）。
  5. 跳转到被调用方法的起始字节码处执行。
  6. 被调用方法执行完毕后，返回调用者栈帧，如果有返回值，将其压到调用者的操作数栈顶。

方法调用的五大指令

JVM规范中定义了五条与方法调用直接相关的字节码指令，每条指令负责一种特定场景的方法调用方式：

调用指令 场景 绑定方式 是否需要实例引用
invokestatic 调用静态方法 静态绑定 不需要
invokevirtual 调用普通实例方法，多态分派 动态绑定(多态) 需要（this）
invokespecial 调用构造器、私有方法、父类方法等 静态绑定 需要
invokeinterface 调用接口方法 动态绑定(多态) 需要（this）
invokedynamic 动态语言支持、Lambda表达式等场景 由Bootstrap指定 视具体场景而定
invokestatic

invokestatic主要用于调用静态方法：

  • 在解析调用目标时，不需要绑定到特定对象实例上，而是直接根据类名和方法描述符来定位静态方法的入口。
  • 由于调用静态方法不需要实例，对应的字节码中也没有“this”引用要传递（相较于invokevirtual需要隐藏传入this参数），所以生成的栈帧也不会包含多余的实例引用参数。

invokevirtual用于调用所有的“实例”方法，除了一些特殊情况（构造器、私有方法和父类方法等，它们使用invokespecial）。invokevirtual会根据对象的实际类型进行动态分派。也就是说，调用的具体方法实现取决于运行时对象的类型，而不是编译时引用类型。

  • 私有方法
  • 使用super关键字调用父类的方法

invokedynamic

invokedynamic是Java 7开始引入的一条新指令，目的是为Java提供一种更灵活的动态调用机制。它并不是只用于调用“动态语言”方法，在Java 8中，invokedynamic也经常被用来实现Lambda表达式、方法引用等功能。与前面四条指令相比，invokedynamic最大的特点在于它不再把方法调用点的解析写死在虚拟机，而是通过“引导方法（Bootstrap Method）”来自定义调用点的解析逻辑。，invokedynamic指令会指向一个CallSite对象。在第一次执行时，JVM会调用对应的引导方法（Bootstrap Method）去确定这个CallSite到底指向哪个具体的方法句柄（Method Handle）。后续执行这个invokedynamic时，就会直接跳转到该句柄所关联的方法，省去了传统的解析过程。在动态语言支持、Lambda实现等场景中，invokedynamic极大地增强了JVM的可扩展性和灵活性。

方法调用的执行流程

在JVM中，每一次方法调用都会触发以下几个流程：

  1. 常量池解析：从当前执行字节码所在类的运行时常量池中，根据索引找到对应的方法符号引用（MethodRef 或 InterfaceMethodRef 等）。如果之前没有被解析过，则需要先进行解析（Resolution），将符号引用转换为直接引用（指向方法元数据的方法指针或方法句柄）。
  2. 检查访问权限：包括类是否可以访问、方法的修饰符（public/private/protected）是否允许，以及与接口方法对应的检查等。如果无法访问或方法不存在，则抛出IllegalAccessError、NoSuchMethodError等异常。
  3. 准备调用参数：如果是实例方法，需要先将对象引用this压入操作数栈顶，然后再按照方法签名依次压入其他参数。对于invokespecial调用的构造器，还需要在调用前先完成对象的初始化等。
  4. 创建新的栈帧：新的方法调用会创建一个新的栈帧，用于存放局部变量表和操作数栈等。参数会从调用者栈帧的操作数栈中弹出，然后存放到被调用方法的栈帧中相应的局部变量表位置（其中this通常放在slot 0的位置）。
  5. 执行方法：从被调用方法的第一条字节码开始，继续解释执行（或者由JIT编译器编译执行）。如果方法里再次调用别的方法，就会重复上述步骤。
  6. 返回结果：当被调用方法执行完成后，如果有返回值，会将返回值压回调用者栈帧的操作数栈顶；如果方法声明为void，则不会返回任何值，但仍然会从栈帧回到上一层调用者的执行位置。

类加载与链接过程

注意一点，方法调用指令引用的方法在字节码文件中通常是一个符号引用。当JVM遇到这些指令时，会尝试去链接 d这个符号引用，找到真正的方法体所在的内存地址或方法句柄。

JVM的类加载和链接过程分为以下几个阶段：

  1. 加载（Loading）：将字节码文件读入到内存，创建Class对象等。
  2. 验证（Verification）：确保字节码文件符合JVM规范，不存在非法代码。
  3. 准备（Preparation）：为类中的静态变量分配内存并设定默认初始值。
  4. 解析（Resolution）：将常量池中的符号引用解析为直接引用。比如：类、字段、方法等的符号引用需要解析为具体的对象或方法指针。如果解析失败，抛出错误。
  5. 初始化（Initialization）：执行类的<clinit>方法，为静态变量赋正确的初始值等操作。

方法调用的优化技术

为了提高方法调用的性能，JVM实现了多种优化技术：

内联缓存

内联缓存是一种在调用点缓存最近调用的方法版本的技术，基于程序运行中的”局部性原理”（即同一个调用点往往反复调用同一个类型的对象）。

方法内联

方法内联是JIT（即时编译器）最重要的优化技术之一，它将被调用的方法体复制到调用点，消除了方法调用的开销。

逃逸分析

逃逸分析确定对象的作用域是否超出方法范围。如果对象不逃逸（仅在方法内使用），可以进行更多的优化：

  • 栈上分配：将对象分配在栈上，而不是堆上，减少GC压力
  • 标量替换：将对象拆分为基本类型变量
  • 同步消除：如果对象不逃逸，可以消除同步锁
