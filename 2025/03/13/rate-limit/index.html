限流算法介绍

发表于 2025-03-13 分类于 架构

在高并发系统设计中，限流是保护系统稳定性的重要手段。当请求量突增时，如果不加以控制，可能导致系统资源耗尽、响应时间变长，甚至系统崩溃。限流算法就像交通信号灯，合理调控请求流量，确保系统平稳运行。

限流算法概述

在介绍具体算法前，先来理解一下限流的基本概念和应用场景。

什么是限流？

限流（Rate Limiting）是指对系统的入口流量进行控制，限制到达系统的并发请求数，保证系统能够正常运行。简单来说，限流就是控制请求的速率，当请求速率超过阈值时，采取相应策略（拒绝、排队、降级等）来保护系统。

为什么需要限流？
  • 防止系统过载：限制请求速率，避免系统资源耗尽
  • 防止恶意攻击：如DDoS攻击
  • 资源分配：合理分配系统资源，优先处理重要请求
  • 服务降级：在峰值时段，通过限流进行服务降级，保证核心功能可用
  • 成本控制：对于按调用次数付费的API，可控制调用频率
常见的限流指标
  • QPS（Queries Per Second）：每秒查询数，衡量系统处理请求的速率
  • 并发数：同时处理的请求数量
  • 总调用次数：限制一段时间内的总调用次数

现在，让我们深入了解三种主要的限流算法。

令牌桶算法（Token Bucket）

基本原理

令牌桶算法是最常用的限流算法之一，其核心思想是系统以一个固定的速率往桶中添加令牌，当请求到来时，需要从桶中获取一个令牌，如果桶中有足够的令牌，则请求被处理，否则请求被拒绝或等待。

想象一下，有一个固定大小的桶，系统按照预设速率（如每秒10个）往桶里扔令牌。当桶满时，新令牌会被丢弃。请求要被处理，必须先获取令牌。令牌不够时，请求要么排队等待，要么被直接拒绝。

令牌桶的特点
  • 可以应对突发流量：因为桶中可能已经积累了一定量的令牌
  • 平滑处理变化的流量：通过调整放入令牌的速率和桶的容量
  • 实现简单，计算量小：适合在高性能系统中使用
  • 可预见性强：可以预先计算出系统能承受的最大峰值

令牌桶算法的应用场景

令牌桶算法适用于需要处理突发流量的场景，例如：

  1. API网关限流：许多API网关如Nginx、Kong等都使用令牌桶算法进行限流
  2. 分布式系统的接口调用：保护下游服务不被突发流量冲垮
  3. 用户行为控制：比如社交媒体发帖频率控制
  4. 云服务资源限制：AWS、GCP等云服务提供商的API调用限制

漏桶算法（Leaky Bucket）

基本原理

漏桶算法的核心思想是水（请求）以任意速率流入漏桶，而漏桶以固定的速率流出水滴（处理请求）。当桶满时，新流入的水将溢出（请求被拒绝）。

这就像一个带有小孔的水桶：无论水流入的速度多快，水流出的速度始终保持恒定。当桶满时，多余的水会溢出（请求被拒绝）。

漏桶的特点
  • 请求处理速率恒定：无论输入流量多大，输出流量始终保持稳定
  • 削峰填谷效果好：能够有效应对流量突增的情况
  • 系统处理更加平滑：对下游系统更加友好
  • 不允许流量突发：与令牌桶不同，即使桶中有空间，也只能按固定速率处理请求

漏桶算法的应用场景

漏桶算法适用于需要稳定输出的场景，例如：

  1. 流量整形（Traffic Shaping）：将突发流量转换为平滑流量
  2. 消息队列处理：控制消息处理速率
  3. 数据库访问控制：以固定速率访问数据库，防止数据库过载
  4. 第三方API调用限制：当第三方服务对调用频率有严格限制时
  5. 下游服务保护：保护下游服务不受流量波动影响

滑动窗口算法（Sliding Window）

基本原理

滑动窗口算法是计数器算法的改进版，它将时间划分为多个小窗口，在每个小窗口内进行计数。当请求到来时，只要当前窗口内的请求计数加上前一个窗口的按比例计算的请求数不超过阈值，请求就会被通过。

想象一个不断移动的时间窗口，窗口内记录了请求的次数。随着时间推移，窗口滑动，新请求进入窗口，旧请求离开窗口。在任何时刻，我们只关心窗口内的请求数量。

滑动窗口的特点
  • 计算简单，实现相对容易：不需要复杂的数据结构
  • 较好的实时性：能够及时反映系统负载情况
  • 内存占用适中：只需存储每个小窗口的计数
  • 有效防止临界问题：避免了固定窗口可能出现的临界问题
  • 精度取决于小窗口的粒度：窗口越小，限流越精确，但计算和存储成本越高

  2. 用户操作频率控制：如登录尝试、验证码发送等

  4. 爬虫控制：限制爬虫爬取频率
  5. 分布式系统中的流量控制：每个节点维护自己的滑动窗口

计数器算法

除了上述三种算法外，还有一种简单但实用的限流算法——计数器算法（也称固定窗口算法）。虽然不是我们重点讨论的对象，但为了比较完整，这里也简单介绍一下。

基本原理

计数器算法的思想非常简单：在一个固定的时间窗口内，维护一个计数器，记录请求的数量。当请求到来时，计数器加一；当计数器达到阈值时，拒绝新的请求；当时间窗口结束时，重置计数器。

计数器算法的特点
  • 实现最简单：只需要一个计数器和一个时间窗口
  • 内存占用最小：只需存储一个计数器和时间戳
  • 存在临界问题：在窗口边界可能出现突增流量

如何选择限流算法？

选择适合的限流算法，需要考虑以下因素：

  1. 流量特性：
    • 流量是否会突发？→ 令牌桶
    • 需要稳定的输出？→ 漏桶
    • 精确控制时间窗口？→ 滑动窗口
  2. 系统资源：
    • 资源充足，计算能力强 → 可选择任何算法
    • 资源受限 → 计数器或简化版令牌桶
  3. 下游服务特性：
    • 下游服务脆弱，需稳定流量 → 漏桶
    • 下游服务强健，可接受突发 → 令牌桶
  4. 实现复杂度：
    • 需要快速实现 → 计数器
    • 可接受一定复杂度 → 令牌桶或滑动窗口
  5. 限流精度要求：
    • 精度高 → 滑动窗口
    • 精度适中 → 令牌桶
    • 精度要求不高 → 计数器
