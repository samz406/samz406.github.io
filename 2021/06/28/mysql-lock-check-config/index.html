MYSQL死锁检测相关配置

使用mysql时总免不了遇到死锁问题，MYSQL的锁全局锁、有表锁、行锁、间歇所锁、next key lock等。MYSQL的锁都是存储引擎自己实现的。比如现在常用的 InnoDB 。支持行锁。下面以行锁为例说明，死锁时该怎么配置。




事务T1 事务T2
begin;
update user set name=’王麻子’ where id=1;		 begin;
update user set name=’王爷’ where id=2;
update user set name=’王麻子’ where id=2;
update user set name=’王爷’ where id=1;

事务 T1 在等待事务 T2释放 id=2 的行锁，而事务T2在等待事务T1释放 id=1 的行锁。 事务T1和事务T2都在等待对方资源释放，所以就发送死锁。那发生死锁以后我们能做些什么呢？mysql为我们提供两种配置策略处理死锁：配置如下

策略1 直接等待

就是直接等待，直到超时。可以通过innodb_lock_wait_timeout 来设置等待时间。 InnoDB 中，innodb_lock_wait_timeout 的默认值是50秒，它表示当发生死锁，第一个被锁住的线程要过50秒才会超时退出，然后其他线程才有可能继续执行。对于越来越没有耐心的人们是很难接受的。那如果把这个时间设置成一个很小的值，比如 2秒。这样当出现死锁的时候，确实很快就可以解开。但是有时候事务比较长，那锁等待就会判断有误。

不足：

​ 直接等待，设置太大或太小都不合适。

策略2 ：发起死锁检测

​ 发起死锁检测是指发现死锁后，主动回滚死锁链条中的某一个事务，让其他事务得以继续执行。设置 innodb_deadlock_detect 参数为 on开启死锁检测。

​ 不足：

​ 在并发量小的情况，死锁检测笔直接等待很划算。每个新来的被堵住的线程，都要判断会不会由于自己的加入导致了死锁，这是一个时间复杂度是 O(n) 的操作。假如有1000个并发线程要同时更新同一行，死锁检测操作就是 100 万量级。虽然最终检测的结果是没有死锁，但是这期间要消耗大量的 CPU资源。现象就是CPU飙升，但是么有几个事务在执行。

那如何解决死锁呢？

​ 那就是写出不会死锁的代码。但是呢，这是很少发生的，所以最好的做法是尽量保证减少死锁，减少锁等待，控制访问相同资源的并发事务量。
