复杂性的定义

杂性是软件系统的主要敌人

复杂性是指任何使软件系统难以理解和修改的结构性因素。

  • 复杂性可能表现为：
    • 理解代码的逻辑或行为困难。
    • 实现简单功能需要大量工作。
    • 修改一个地方导致其他地方出错。
  • 一个系统的复杂性与其规模或功能无关，即使是小型系统也可能非常复杂。

复杂性可以从开发成本的角度理解：在复杂系统中，哪怕是微小的改动也可能需要巨大代价；而在简单系统中，大改动所需的努力也较少。

复杂性的三大表现

复杂性主要通过以下方式显现：

（1）变化放大（Change Amplification）

一个看似简单的改动会要求对多个地方进行修改。

  • 示例：在早期网站中，每个页面都直接硬编码了背景颜色。修改颜色需要手动更新所有页面，这种设计增加了修改代价。
  • 改进方式：集中式管理，例如通过共享变量统一颜色定义，从而减少需要修改的地方。

（2）认知负担（Cognitive Load）

开发者需要记住或学习大量信息才能完成任务。

  • 示例：一个函数分配了内存，并要求调用方释放。这种隐含责任增加了开发者的记忆负担，容易导致错误（如内存泄漏）。
  • 改进方式：将资源的分配和释放责任封装在函数内部，降低调用方的认知负担。

（3）未知的未知（Unknown Unknowns）

开发者在完成任务时，不知道需要修改哪些地方，也不清楚可能涉及的依赖或隐含规则。

  • 示例：某些页面的颜色强调部分依赖于背景颜色，但开发者未被告知，导致修改背景颜色时忽略这些依赖，从而引发错误。
  • 特点：未知的未知是最危险的复杂性表现，因为开发者甚至意识不到问题的存在。

复杂性的两大成因

复杂性主要源于以下两种因素：

（1）依赖性（Dependencies）

  • 定义：当代码无法独立存在，需与其他代码协作时，就会引入依赖。
  • 示例：
    • 不同模块之间共享某个全局变量。
    • 函数签名的变更要求更新所有调用者。
  • 影响：依赖性会导致变化放大和高认知负担，使系统更难维护。
  • 解决方式：通过模块化设计和接口隐藏，减少模块间的依赖性。

（2）晦涩性（Obscurity）

  • 定义：当系统中的关键信息无法直观获取时，会产生晦涩性。
  • 示例：
    • 变量名过于模糊（如 time），开发者难以理解其用途。
    • 文档缺乏详细说明，开发者不得不阅读源码才能理解设计意图。
  • 影响：晦涩性会导致未知的未知，增加开发难度。
  • 解决方式：通过清晰的命名、良好的文档和一致的设计，降低晦涩性。

依赖性和晦涩性常常共同作用，导致复杂性的各种表现。例如，当多个模块依赖于未显式说明的信息时，晦涩性会加剧依赖的危害。

复杂性的增量累积

复杂性往往以微小的方式逐步累积，形成难以控制的局面。

  • 积累过程：

    • 每次改动引入少量依赖或晦涩性。
    • 当积累达到一定程度，系统变得难以维护，哪怕是小的改动也会触发连锁反应。

  • 应对策略：

    • 采取“零容忍”态度，避免复杂性的每一次不必要增加。
    • 识别复杂性“红旗”（如重复代码、隐藏依赖），在每次改动中消除潜在问题。

设计的核心目标

优秀的系统设计应：

  • 减少依赖性：使模块之间尽可能独立，避免多处修改。

  • 降低晦涩性：通过清晰的接口和一致的设计，使系统更直观。

  • 最小化复杂性的表现

    • 减少需要修改的代码量（对抗变化放大）。
    • 降低开发者的记忆负担（对抗认知负担）。
    • 消除隐含规则或未显式的依赖（对抗未知的未知）。

在《Software Architecture Patterns》一书中介绍了5种架构设计模式，分别是分层、事件驱动、微内核、微服务、基于空间架构模式。下面分别介绍下每种设计模式

分层架构

​ 最常见的架构模式是分层架构模式，也称为n层架构模式。工作多年几乎都是采用分层架构。

描述

​

如上图所示，分层架构模式中的组件按照水平分层，每一层在其中扮演特定的角色 (如表示逻辑或业务逻辑)。尽管 分层架构模式不指定分层类型，但通常是由四个标准层组成:表示层、业务层和持久性层 和数据库。在某些情况下，业务层 和持久层合并成一个业务层。

分层架构模式的每一层都有特定的作用。例如，展现层将负责处理所有用户界面和浏览器通信逻辑，而业务逻辑层负责处理业务逻辑。每一层为满足特定业务都形成了一个抽象。例如，展现层不需要知道 或者担心如何获取客户数据；它只需要以特定格式显示在屏幕上。同样， 业务逻辑层无需关心数据如何显示在屏幕上，它只需要从持久化中获取数据，对数据执行业务逻辑，并将该信息传递给上层。

​

如上图所示，架构中的每一层都是标记为关闭。 这是分层中一个非常重要的概念。closed层意味着作为请求从一层移到另一层，也就是一层一层的往下调用。 例如，一个来自展现层的请求必须先经过业务逻辑层，然后到持久层，最后到达数据库层。

为什么不允许表示层直接访问持久层or 数据库层？ 毕竟直接数据库访问从表示层开始比通过一堆不必要的层只是为了从数据库获取数据或将数据保存到数据库。 主要是因为隔离层。隔离层概念意味着在一个层中不会影响其他层，更改层内组件，可能相关的层被修改。 如果允许展现表示层访问持久层，然后在持久层会影响业务层和展现层，这样就会出现一个混乱的依赖关系，单项变为双向依赖或其他。 这样的后果就是架构变更非常难，成本也很高。隔离层的概念也意味着每一层都是独立与其他层，从而很少或根本不知道其他层的内部工作。

事件驱动架构

事件驱动架构模式是一种基于异步、可扩展性分布式架构模式。 它可用于小型应用程序以及大型复杂的应用程序。

事件驱动架构模式由两个主要拓扑组成：中介和brocker。中介作用于一个事件当需要编排多个步骤时，而brocker拓扑是当您不使用中介的情况下将事件链接在一起时。 因为架构特点和实现这两种拓扑的连接策略不同，所以需要了解每一个模式，这样才能选一个最适合你的场景。

顺序消息

消息有序指的是一类消息消费时，能按照发送的顺序来消费。例如：一个订单产生了三条消息分别是订单创建、订单付款、订单完成。消费时要按照这个顺序消费才能有意义，但是同时订单之间是可以并行消费的。RocketMQ可以严格的保证消息有序。

顺序消息分为全局顺序消息与分区顺序消息，全局顺序是指某个Topic下的所有消息都要保证顺序；部分顺序消息只要保证每一组消息被顺序消费即可。

  • 全局顺序
    对于指定的一个 Topic，所有消息按照严格的先入先出（FIFO）的顺序进行发布和消费。
    适用场景：性能要求不高，所有的消息严格按照 FIFO 原则进行消息发布和消费的场景

  • 分区顺序
    对于指定的一个 Topic，所有消息根据 sharding key进行区块分区。 同一个分区内的消息按照严格的 FIFO 顺序进行发布和消费。 Sharding key 是顺序消息中用来区分不同分区的关键字段，和普通消息的 Key 是完全不同的概念。
    适用场景：性能要求高，以 sharding key 作为分区字段，在同一个区块中严格的按照 FIFO 原则进行消息发布和消费的场景。

消息重试

Consumer消费消息失败后，要提供一种重试机制，让消息再消费一次。Consumer消费消息失败通常可以认为有以下几种情况：

  • 由于消息本身的原因，例如反序列化失败、消息数据本身无法处理（例如话费充值，当前消息的手机号被注销，无法充值）等。这种错误通常需要跳过这条消息，再消费其它消息，而这条失败的消息即使立刻重试消费，99%也不成功，所以最好提供一种定时重试机制，即过10秒后再重试。
  • 由于依赖的下游应用服务不可用，例如db连接不可用，外系统网络不可达等。遇到这种错误，即使跳过当前失败的消息，消费其他消息同样也会报错。这种情况建议应用sleep 30s，再消费下一条消息，这样可以减轻Broker重试消息的压力。
